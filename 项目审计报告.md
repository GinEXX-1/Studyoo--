## 高优先级问题（必须在开工前解决）

### 1. 追问功能设计与契约不一致

位置：`AI学习网站 API契约 v1.0.md` 第 3.5 节、`我与 Claude 的对话.md` 末尾

对话中已明确决定：追问改成 **A/B/C/D 选项式**（A/B/C 根据题目生成，D 是自定义提问），避免 AI 侵占主思路。但契约第 3.5 节仍是旧设计——开放式 `content_text` 进、`reply_text` 出。

风险：Codex 和 Trae 按旧设计开发，最终做出的追问是聊天气泡，违背你确定的核心原则。

建议：将追问接口改为：

```json
// GET /questions/{question_id}/follow-up-options
// 先获取系统生成的追问选项
{
  "success": true,
  "data": {
    "options": [
      { "id": "A", "text": "这一步为什么这样做" },
      { "id": "B", "text": "有没有更简单的方法" },
      { "id": "C", "text": "这个知识点在教材哪一页" }
    ],
    "allow_custom": true
  },
  "message": "ok"
}

// POST /questions/{question_id}/follow-up
// 用户选择选项或输入自定义问题
{
  "option_id": "A" | null,
  "custom_text": "string | null"
}

// 响应
{
  "success": true,
  "data": {
    "reply_text": "..."
  },
  "message": "ok"
}
```

### 2. 技术栈未确定

位置：未在任何文档中明确

契约第 6 节定了运行时约束，但未指定技术栈：

- 后端框架：Node.js/Express 还是 Python/FastAPI？
- 数据库：PostgreSQL、MySQL 还是 Supabase？
- 前端框架：React + Vite、Next.js 还是其他？

风险：Codex 和 Trae 各自选择不同技术栈，导致无法协作。

建议：第一版建议选择简单组合：
- 后端：Python/FastAPI + PostgreSQL（生态成熟，AI 对接方便）
- 前端：React + Vite（轻量，社区成熟）

### 3. CORS（跨域资源共享）未提及

位置：`AI学习网站 API契约 v1.0.md` 第 0.1 节

前端开发环境 `http://localhost:5173`，后端 `http://localhost:3000/api/v1`，不同源必然涉及 CORS。契约未说明：

- 允许哪些来源（origin）
- 允许哪些 HTTP 方法
- 是否允许携带 Authorization 头

风险：前端请求被浏览器拦截，联调时出现"跨域错误"。

建议：在契约第 6 节增加：
- 开发环境允许 `http://localhost:5173`
- 生产环境允许 `https://<your-domain>`
- 允许 GET/POST/PATCH/DELETE
- 允许 Authorization 头

### 4. PATCH 接口缺少所有权鉴权规则

位置：`AI学习网站 API契约 v1.0.md` 第 4.2、5.2 节

`PATCH /mistakes/{mistake_id}` 和 `PATCH /learning-path/{item_id}` 未说明"只能修改自己的记录"。

风险：用户 A 可以修改用户 B 的错题状态或学习路径推荐。

建议：在契约中明确：所有 PATCH/DELETE 接口必须校验资源所有权，只有资源所属用户才能操作。

### 5. knowledge_tags 来源与归一化未定义

位置：Question、MistakeRecord、LearningPathItem 模型

多个模型依赖 `knowledge_tags`，但未说明：
- 标签由谁生成（AI 自由生成还是有标准字典）
- 是否需要归一化（如"排列组合""排列与组合""计数原理-排列组合"应统一）

风险：同一知识点出现多种写法，统计失真，知识图谱无法构建。

建议：第一版至少建立轻量标准：
- 学科固定枚举：数学、物理、化学、生物、语文、英语、历史、地理、政治
- 知识点标签允许 AI 生成，但后端需做归一化（同义词映射）
- 数据库保留原始标签和规范标签，或先维护一份简单标签字典

## 中优先级问题（建议在第一阶段解决）

### 6. error_rate 语义错位

位置：`AI学习网站 API契约 v1.0.md` 第 4.3 节

字段名为 `error_rate`，但错题判定规则是"用户点击过'直接看答案'即计入"。这不是"错误率"，是"求助率"。

风险：薄弱点统计把"好奇看答案"行为算成错误，推荐结果偏差。

建议：字段名改为 `help_rate`（求助率），或保持 `error_rate` 但在产品文案中明确说明统计口径。

### 7. revealed_full_solution 在 deepen 模式下语义不明

位置：`AI学习网站 API契约 v1.0.md` 第 1.3、3.3 节

Answer 模型有 `revealed_full_solution` 字段，但 3.3 说明 deepen 模式直接返回 `full_solution_text`。此时该字段应为 true、false、还是 null？

风险：前端根据该字段做展示逻辑，可能出现不一致。

建议：明确约定：
- `solve_from_scratch` 模式：初始为 false，用户请求完整解答后变为 true
- `deepen_official_answer` 模式：始终为 true（因为答案直接返回）

### 8. token 存储方式不明确

位置：`AI学习网站 API契约 v1.0.md` 第 0.2 节

契约写"前端存储在内存或安全存储中"，但未说明具体方案。

风险：前端可能使用 localStorage（XSS 可读取），token 泄露。

建议：明确推荐方案：
- 使用 HttpOnly cookie（安全性最高）
- 或内存存储 + 短时间过期（用户刷新需重新登录）

### 9. AI 超时无前端 UX 指引

位置：`AI学习网站 API契约 v1.0.md` 第 6 节

第 6 节说明 AI 请求超时 60 秒，但未说明超时后前端如何展示。

风险：用户看到白屏或无限转圈，体验差。

建议：明确超时后的用户体验：
- 显示加载动画，标注"AI 正在思考..."
- 超时后显示"AI 暂时没回应，请稍后重试"
- 返回 `AI_SERVICE_ERROR` 错误码

### 10. README 与契约的"唯一真相来源"表述冲突

位置：`README.md` Development Principles、`AI学习网站 API契约 v1.0.md` 开头

README 写"README 是整个项目唯一可信的设计文档"；契约写"本文档是前后端协作的唯一真相来源"。

风险：需求变更时不知道先改哪个文档。

建议：明确分层：
- README：产品愿景、定位、路线图
- API 契约：前后端接口唯一真相来源
- PROGRESS.md：当前实现状态唯一真相来源

## 低优先级问题（可在开发过程中逐步完善）

### 11. PROGRESS.md 责任重复

位置：三份 agent 分工文档

Codex、Trae、CodeBuddy 都被要求维护进度，但未规定最终合并者。

建议：只保留一份根目录 `PROGRESS.md`，由 CodeBuddy 作为最终维护者；Codex 和 Trae 每次完成任务时提供"可验证状态"，由 CodeBuddy 合并。

### 12. AI 输出质量无可测试验收标准

位置：`Codex任务分工——后端.md` 第 2.1、2.2 节

文档强调"讲透""不要直接给答案"，但没有固定测试题集。

建议：建立 `验收题集.md`，至少包含：
- 一道含分式/根号/求和符号的数学题（测公式渲染）
- 一道官方答案跳步的数学题（测 `deepen_official_answer`）
- 一道从零不会的题（测 `solve_from_scratch`）
- 一道追问（测上下文保留）

### 13. README 愿景与契约范围错位

位置：`README.md` Core Capabilities、`AI学习网站 API契约 v1.0.md`

README 描绘的是"Learning OS"（7 大能力），契约实际覆盖的只是 v0.1 的小型 Web App（答疑 + 错题本 + 学习路径）。

风险：agent 会拿 README 的宏大范围去理解契约的小 scope，造成范围蔓延。

建议：在 README 中明确标注"当前实现范围：v0.1"，避免误解。
