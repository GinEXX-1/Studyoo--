# Studyoo 项目审计报告

审计日期：2026-07-05

## 结论

当前项目还没有实际代码，仓库内容是产品定位、API 契约与三方 agent 分工文档。因此本次审计对象是“项目是否已经具备开工条件”，不是代码质量。

总体判断：方向清楚，核心差异化明确，文档比常见早期项目更完整；但仍不建议立即让三个 agent 并行开工。原因是 API 契约里还有若干未拍板事项，以及部分接口响应形状不够统一。若不先收口，后续仍可能重演“前后端各做各的、最后连不上”的问题。

## 高优先级问题

### 1. 关键技术决策仍未拍板

位置：`AI学习网站 API契约 v1.0.md` 第 6 节、`Codex任务分工——后端.md` 第 1.3 节

未决定项包括：

- AI 服务商、模型、超时时间、重试次数
- 图片存储方案
- 单用户每日限流规则
- token 过期时间与过期处理方式

风险：Codex 无法稳定实现后端，Trae 也无法正确处理超时、失败、重新登录、图片上传等流程。

建议：开工前把第 6 节从“尚未决定”改成“已决定”。第一版可以保守选择：

- AI 调用：单一模型供应商，先不做多供应商切换
- 超时：60 秒
- 重试：AI 生成失败不自动重试，网络错误最多重试 1 次
- 图片：第一版不持久化原图，只保存 OCR 后文本；如需回看原图，再引入对象存储
- 限流：每用户每天 30 次 AI 请求
- token：7 天过期，过期后重新登录

### 2. `POST /questions` 的响应结构前后描述冲突

位置：`AI学习网站 API契约 v1.0.md` 第 3.1、3.1b、3.3 节

第 3.1 和 3.1b 写的是提交问题后“响应 Question 对象，status 为 pending”；第 3.3 又建议第一版采用同步方案，让 `POST /questions` 阻塞等待 AI，并在响应中一并返回 Answer。

风险：后端可能只返回 Question，前端却期待 `{ question, answer }`；或者前端做轮询，后端却是同步返回。

建议：第一版明确采用同步方案，并统一写死：

```json
{
  "success": true,
  "data": {
    "question": {},
    "answer": {}
  },
  "message": "ok"
}
```

同时把 Question 的 `status` 约定改清楚：同步成功时为 `answered`，AI 失败时为 `failed`，不要默认写 `pending`。

### 3. 统一响应格式没有贯穿所有接口示例

位置：`AI学习网站 API契约 v1.0.md` 第 2-5 节

全局约定要求所有接口返回 `{ success, data, message }`，但后面许多接口示例只写了 `data` 内部结构，例如 `{ "reply_text": "..." }`、`{ "tags": [...] }`。

风险：前端请求层会按统一响应解析，但某些接口实现者可能照局部示例直接返回裸对象。

建议：所有接口示例都改成完整响应包裹。尤其是：

- 登录/注册
- 提交问题
- 追问
- 错题列表
- 薄弱点统计
- 学习路径列表

### 4. 图片输入与图片存储策略互相打架

位置：`AI学习网站 API契约 v1.0.md` 第 1 节 Question 模型、第 3.2 节、第 6 节

Question 模型包含 `content_image_url` 和 `official_answer_image_url`；第 3.2 又建议第一版不单独做图片存储；第 6 节仍要求拍板图片存储。

风险：数据库字段、前端展示、后端上传流程会各自理解不同。

建议二选一：

- 最简方案：第一版不保存图片 URL，图片只用于当次 OCR/多模态识别，模型字段保留但固定返回 `null`
- 可回看方案：接入对象存储，明确 URL 是否永久、是否签名、是否过期

如果选择最简方案，契约里要明确写：`content_image_url` 第一版恒为 `null`。

### 5. Base URL 可能与前端开发端口冲突

位置：`AI学习网站 API契约 v1.0.md` 第 0.1 节

开发环境写成 `http://localhost:3000/api/v1`。如果 Trae 使用 Vite/React，前端常见端口是 5173；如果 Next.js，3000 可能是前端服务本身。

风险：前端和后端抢同一个端口，或开发代理配置混乱。

建议：先定技术栈后再定端口。若后端独立服务，建议：

- 前端：`http://localhost:5173`
- 后端：`http://localhost:3000/api/v1` 或 `http://localhost:8000/api/v1`

并在前端文档中要求通过环境变量配置 API 地址。

## 中优先级问题

### 6. README 与 API 契约的“唯一真相来源”表述不一致

位置：`README.md` Development Principles、`AI学习网站 API契约 v1.0.md` 开头

README 写“README 是整个项目唯一可信的设计文档”；API 契约写“本文档是前后端协作的唯一真相来源”。

风险：需求变更时不知道先改 README 还是 API 契约。

建议：明确分层：

- README：产品愿景、定位、路线图
- API 契约：前后端接口唯一真相来源
- PROGRESS.md：当前实现状态唯一真相来源

### 7. 认证方案过于简略

位置：`AI学习网站 API契约 v1.0.md` 第 0.2、2 节

目前只写 nickname + password，没有提密码哈希、重名规则、密码长度、token 类型。

风险：后端实现可能出现明文存储密码、昵称冲突、弱密码等问题。

建议补充：

- 密码必须哈希存储
- nickname 是否唯一
- password 最小长度
- JWT secret 使用环境变量
- 登录失败统一错误，避免泄露用户是否存在

### 8. 错题判定规则太粗，但可接受，需要标注副作用

位置：`AI学习网站 API契约 v1.0.md` 第 4.3 节

“点击直接看答案即计入错题本”适合第一版，但它并不等价于真的做错。

风险：薄弱点统计会把“好奇看答案”的行为也算作错误，推荐结果可能偏差。

建议：第一版保留该规则，但 UI 文案不要叫“错误率”，可改成“求助率”或“直接看答案比例”。如果仍使用 `error_rate` 字段，至少在产品文案里降低绝对判断感。

### 9. 知识点标签来源未定义

位置：Question、MistakeRecord、LearningPathItem 模型

多个模型依赖 `knowledge_tags`，但没有说明标签由谁生成、是否允许 AI 自由生成、是否有标准标签表。

风险：同一知识点可能出现“排列组合”“排列与组合”“计数原理-排列组合”等多个写法，统计会失真。

建议：第一版至少建立一个轻量标准：

- 学科固定枚举
- 知识点标签允许 AI 生成，但后端需要做归一化
- 数据库保留原始标签和规范标签，或先维护一份简单标签字典

### 10. `PROGRESS.md` 责任重复

位置：三份 agent 分工文档

Codex、Trae、CodeBuddy 都被要求维护进度，但没有规定谁最终负责合并。

风险：出现多份进度、状态互相矛盾。

建议：只保留一份根目录 `PROGRESS.md`，CodeBuddy 是最终维护者；Codex 和 Trae 每次完成任务时必须提供“可验证状态”，由 CodeBuddy 或用户合并进进度文档。

## 低优先级问题

### 11. 视觉规范较细，但缺少可执行页面清单

位置：`AI学习网站 API契约 v1.0.md` 第 7 节、`Trae 任务分工 —— 前端.md`

视觉规范很完整，但还缺少第一版页面范围冻结。

建议第一版只做：

- 登录/注册
- 首页提问
- 答案展示
- 追问
- 错题列表
- 薄弱点统计
- 学习路径列表

不要在 v0.1 做完整仪表盘或复杂 Workspace。

### 12. AI 输出质量验收还不够可测试

位置：`Codex任务分工——后端.md` 第 2.1、2.2 节

文档强调“不要直接给答案”“讲透官方答案”，但还没有固定测试题集。

建议建立 `验收题集.md`，至少包含：

- 一道分式/根号题，测公式渲染
- 一道官方答案跳步的数学题，测 `deepen_official_answer`
- 一道从零不会的题，测 `solve_from_scratch`
- 一道追问，测上下文是否保留

## 建议的开工顺序

1. 先修改并锁定 API 契约第 6 节。
2. 统一 `POST /questions`、`GET /questions/{id}/answer` 的同步/异步策略。
3. 建立根目录 `PROGRESS.md` 模板。
4. 再让 Codex 搭后端最小链路：注册、登录、提交一道文字题、真实 AI 返回、写入数据库。
5. 后端最小链路跑通后，Trae 再做首页提问和答案展示。
6. CodeBuddy 做第一轮端到端验收。

## 是否可以开工

不建议直接全面开工。

建议先做“契约收口 + 最小真实链路”，目标是 1-2 天内跑通：

注册登录 -> 提交一道文字数学题 -> AI 返回解析 -> 数据库存 Question/Answer -> 前端展示结果。

这条链路跑通后，再扩展图片、错题本和学习路径。这样最符合本项目最重要的原则：不做看起来能跑、实际是假数据的前端 MVP。
