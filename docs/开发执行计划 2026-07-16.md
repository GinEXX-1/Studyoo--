日期：2026-07-16
依据：《项目审计与开发方向 2026-07-16》及当前代码复核

## 判断

Claude 的主判断成立：真实学生验证是前置闸门；评阅等待、PDF 长任务和线上可观测性是内测阶段最值得处理的工程风险；硬编码学习图谱不能继续包装成完整知识树。

两处执行修正：

1. 流式评阅采用一次 AI 调用。模型流式返回 JSON，前端增量显示 `feedback_text`，流结束后对同一份完整 JSON 做 schema 校验和入库，避免双调用产生矛盾与双倍成本。
2. 已完成的网页爬虫保留为管理员供给工具，但共享题库降为二级入口。爬虫候选只有人工审核通过后才发布。

## S0：内测闸门（现在）

唯一硬指标：至少 3 名真实学生各自导入一张最近的试卷，完整走完“导入 → 做题 → 评阅 → 订正 → 重做”，并留下漏斗数据与访谈记录。

工程侧只允许三类工作：修真实用户遇到的问题；完成 S1/S2 的可靠性工作；清理会污染内测结论的入口和文案。暂停新的一级导航、新社区机制和全科知识图谱展示。

验收材料：`npm run report:funnel --workspace backend` 输出、每位学生的任务完成记录、15 分钟访谈笔记。

## S1：评阅体验与可观测性（3-4 天）

状态：已完成，等待随首轮真实学生使用继续观察线上错误率。

1. 单次调用的 SSE 流式评阅；缓存命中直接返回完成事件；断流时前端降级到原同步接口。
2. 结构化请求日志，至少包含 request_id、路径、状态码、耗时；AI 日志包含模型、耗时、token 和错误类别，绝不记录密码、Cookie、完整作答或 API Key。
3. CI 覆盖 `main`、`agent/**`、`claude/**`，并纳入爬虫测试。

验收：首个进度事件小于 300ms；真实 AI 首段反馈目标小于 3 秒；任何 5xx 可用 request_id 在 Railway 日志定位。

## S2：PDF 导入可靠性（按内测结果提级）

源码复核显示，Claude 审计所说的“20 页视觉识别仍在单个 HTTP 请求中”已部分过时：当前前端先调用 `process-all?prefer_text_only=1` 做本地文字层探测；需要视觉模型时，按页面逐个调用 `/pages/:pageId/process`，每页结果都会落库并更新进度。

剩余风险是：PDF 上传和 `pdftoppm` 渲染仍在同步请求内；刷新后前端没有继续未完成任务的入口；后端仍保留不带 `prefer_text_only` 的旧整批视觉路径。首轮内测若在导入阶段出现超时或刷新丢进度，则立即按以下顺序处理：

1. 前端增加“继续上次导入”，从 `GET /import/pipeline/tasks` 和页面状态恢复；
2. 禁止客户端调用旧整批视觉路径，只保留文字层快速路径和单页处理；
3. 上传/渲染改为持久化后台任务，必要时再由 base64 JSON 升级为 multipart。

验收：视觉识别始终按页请求；进度可见；处理中刷新页面可继续；重复处理同一页不重复生成候选；20 页 PDF 的上传渲染不触发平台超时。

## S3：数据驱动修复与分层（首轮内测之后）

先按真实漏斗关闭问题，再依次抽取 repositories、services 和路由。每一步保持行为不变并运行全套守护栏。目标不是一次性重写，而是让新增功能不再继续扩大巨石文件。

## 知识树立项

知识树属于教研数据项目。第一阶段只做数学，目标 100-200 个经过人工校验的节点，包含父子关系、前置依赖、规范标签映射和课标来源。完成前，学生端只展示“已练知识点能力地图”，不得宣称已有完整课程知识树。

认知模型 v1 的启动条件固定为：首轮内测后能够计算 7 日留存，且 7 日留存达到 40%。届时以“学生 × 规范知识点”为状态单元，用每次评阅、订正、重做和间隔复习作为观测更新记忆状态。在指标达到前，不新增模拟认知模型的可视化。
