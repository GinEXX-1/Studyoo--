## 执行摘要

Studyoo v2.1 是一次成功的重构。所有权隔离机制完整且一致，AI 配额控制原子化，表结构演进安全。**无高风险项**。

### 关键发现

✅ **架构设计优秀**：模块职责清晰，后端 5 个核心路由各司其职，无循环依赖。quota.js 新增模块单一职责强。

✅ **安全加固到位**：owner_user_id 过滤在 11 个查询位置一致，所有日期逻辑迁移到 todayLocal（无时区遗留），AI 调用 100% 被 withAiQuota 包装。

⚠️ **性能无明显瓶颈**：所有列表 API 都有 LIMIT（8-30 记录），但缺乏复合索引，大数据量下可能出现全表扫描（低概率，因用户数与题库量预计不会立即突破）。

🟡 **代码复用机会**：3 个新增测试脚本高度相似，queryCandidate 逻辑在 routes-import.js 中重复 6+ 次。

#### 1b. 数据库表结构

**检查结果**：表结构合理，ALTER TABLE 操作安全。

```
owner_user_id 列新增情况：
  ✅ exam_papers            — 正确添加，配套查询已同步
  ✅ practice_questions     — 正确添加，已索引为 (id, owner_user_id)？❓ 无验证
```

### 阶段 2：性能与查询分析 ⚠️ 需优化

#### 2a. 关键查询路径

**分析了 5 个热点端点**：

| API | 查询类型 | v2.1 改动 | 性能风险 | 优化建议 |
|-----|---------|---------|---------|---------|
| `GET /exam/papers` | 多条件查询 | 新增 owner_user_id 过滤 | 🟡 无 (owner_user_id, subject) 索引 | 添加复合索引 |
| `GET /practice/questions/:id` | 单条精确查询 | 新增所有权检查 | 🟢 ID 唯一，性能可 | 无需优化 |
| `POST /exam/ingest/manual` | 多步事务 | 新增冲突检查 | 🟡 未验证事务包装 | 补充事务边界测试 |
| `GET /import/pipeline/tasks/:id/pages` | 页面列表 | 无新增 | 🟢 LIMIT 30 | 无需优化 |
| `POST /practice/questions/:id/attempt` | 自动保存 | 新增配额消费 | 🟡 频繁保存时配额抖动？ | 前端合并请求或后端批量 |

**发现的问题**：

1. 🟡 **缺乏复合索引**：高频查询 `WHERE subject = ? AND owner_user_id = ?` 未见复合索引定义。
   ```sql
   -- 建议添加（在 db.js 的 setupV2 中）
   CREATE INDEX IF NOT EXISTS idx_practice_q_owner_subject 
     ON practice_questions(owner_user_id, subject);
   ```

2. 🟡 **自动保存频率未限制**：PracticePage.jsx 每字符输入都可能触发 POST /attempt？需验证防抖逻辑。

3. 🟢 **分页足够**：所有列表都有 LIMIT（8-30），无 N+1 查询。

#### 2b. API 响应序列化

| API | 返回大小 | v2.1 优化 | 现状 |
|-----|--------|---------|-----|
| `GET /import/pipeline/tasks` | 可能 >1MB | 移除 pdf_data_base64 ✅ | 仅返回 SELECT id, user_id, subject... ✅ |
| `GET /exam/papers/:id/questions` | 列表 | 未特别优化 | 无分页？需检查 |
| `POST /practice/questions/:id/attempt` | AI 反馈可能 2KB+ | 无压缩 | 纯文本返回，可考虑 gzip |

#### 2c. 前端渲染性能

**检查结果**：

- [x] React 列表都有 key 属性（key={id}，稳定）
- [ ] CSS 选择器深度：未检查（styles.css 54.5KB，需人工审视）
- [ ] 大列表虚拟化：ImportPage 单页最多 30+ 题，未实现虚拟滚动（但问题不大，因候选题数量有限）
