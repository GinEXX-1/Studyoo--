
你负责整个项目的前端页面与交互。所有接口调用必须严格对照 `API契约.md`，视觉呈现必须严格对照契约第7节的视觉设计规范。你不负责后端逻辑、数据库、AI调用——那是Codex的范围，不要自己在前端写"临时用的假逻辑"去绕过后端。

**最重要的一条规则：严禁使用mock data。** 如果某个接口Codex还没做完，就先把这部分功能标注"待接入"并暂停开发，不允许用假数据填充继续往下做页面。这是上一版项目失败的直接原因，这次绝对不能重犯。

## 1. 全局基础设施（最先做）

### 1.1 视觉基础

按契约7.1-7.4：

- 引入 `Noto Serif SC`（或类似中文衬线字体）用于标题、题目内容、公式讲解正文
- UI控件（按钮、标签、导航）用无衬线体，这个区分要做成组件库层面的规则，不要每个页面自己判断
- 建立全局色彩变量：主强调色蒂芙尼系（`#7FC9C1` / `#C7EAE6` / `#1F5D57`）、副强调色赤陶系（`#A85C42` / `#F0DED2` / `#FCF3EC`）、背景 `#FCF9E8`
- 页面背景纹理按契约7.3给的CSS实现，只用在页面级背景，卡片内部保持纯色

### 1.2 统一请求层

封装一个统一的API请求函数，处理：

- 自动带上 `Authorization: Bearer <token>` 请求头
- 统一解析 `{ success, data, message, error_code }` 响应格式，成功/失败只看 `success` 字段，不看HTTP状态码
- token失效（`AUTH_INVALID_TOKEN`）时统一跳转登录页

不要每个页面各自写一套请求逻辑，否则以后错误处理方式会到处不一致。

### 1.3 数学公式渲染组件（重要，必须复用）

- 引入 KaTeX（不要用MathJax，太重）
- 实现一个统一组件：输入一段可能包含 `$...$` / `$$...$$` 的文本，扫描并分段渲染成公式+普通文字混排
- 所有展示AI讲解内容的地方（答疑结果、追问回复、错题本详情），一律复用这一个组件，不允许每个页面自己写一遍公式解析逻辑

## 3. 首页与提问界面（对应第3-5周，产品核心，重点打磨）

### 3.1 首页

参考已确认的视觉方向：首页就是提问入口，不做仪表盘，保持最直接的焦点。核心元素：

- 大标题（衬线体）："遇到不会的题，来问问看"类似文案
- 模式切换：「从零开始问」/「深化官方答案」两个选项，用蒂芙尼色标注选中状态
- 输入区域：`solve_from_scratch` 模式只需一个题目输入框；`deepen_official_answer` 模式必须有**两个独立输入框**（题目 / 官方答案），不能合并成一个大文本框——这个区分在契约里特别强调过，因为AI需要明确区分哪部分是题目哪部分是官方答案
- 提交按钮用赤陶色（全站的"行动色"，不要在别的地方乱用这个颜色）

### 3.2 图片输入

支持拍照/上传图片作为题目输入，`deepen_official_answer` 模式下官方答案也支持图片或文字二选一。

### 3.3 答案展示页

两种模式展示逻辑不同，务必分开处理：

- `solve_from_scratch`：先展示 `hint_text`，用户点击"看完整答案"按钮才请求并展示 `full_solution_text`
- `deepen_official_answer`：直接展示 `step_breakdown`（做成可展开的分步骤卡片，每步一个编号），不要藏起来——这个模式下用户已经有答案了，"卖关子"没有意义
- 追问输入框，追问回复同样要经过数学公式渲染组件处理

验收标准：真实提交一道包含公式的题目，两种模式各测一遍，确认公式渲染正常、模式差异符合预期。

## 6. 每完成一个页面，必须做的事

1. 对照 `API契约.md` 联调进度表，把"前端对接完成"和"真实数据联调通过"两列打勾——注意这是两件事，接口接上了不代表联调通过，要真的在浏览器里操作一遍看到正确结果才算
2. commit信息用中文写清楚：做了什么页面/组件、对接了哪个接口、有没有已知的待办
3. 遇到接口返回的数据和契约文档不一致，先跟Codex确认是谁的理解有误，不要自己在前端加一层"猜测性的兼容处理"糊弄过去
