# Studyoo 优化重构升级方案

制定日期:2026-07-15
依据:README、PROGRESS.md、《战略审计与方向 2026-07-13》、《下一阶段功能计划 2026-07-14》、前后端全部源码(backend/src 约 4900 行,frontend/src 约 4400 行)

---

## 一、项目立意的亮点

### 1. 反共识的定位,切中了真实空白
"Parse, Don't Answer / AI 不替代思考"不是一句口号,而是贯穿到了实现里:solve_from_scratch 模式先给思路引导不给答案、reveal-solution 需要学生主动请求且同步计入错题本、评阅先指出做对的地方再指出缺口。在搜题(作业帮/小猿)和刷题 App 的红海里,"训练理解"而非"投喂答案"是目前少有人认真做的方向,且与家长/老师的诉求天然一致。

### 2. 第一场景选对了,闭环是独有的
"导入自己的试卷 → AI 拆题入库 → 独立作答 → AI 评阅 → 订正 → 薄弱点沉淀"这条链路,市面产品没有完整给出。学生自己的月考卷是独占数据资产——供给侧不同质化,且用得越久个人题库和错题数据越厚,天然有留存壁垒。《战略审计》里"别的 App 帮你看懂题,Studyoo 逼你做对它"这句定位是准确的。

### 3. AI 工程化细节有真功夫
这些细节远超同等规模的业余项目水准:
- 评阅提示词带**评分锚点**(90-100/80-89/…,is_correct=score≥80),抑制忽严忽宽;
- 知识点标签做了 **canonical 归一化 + 别名表**,避免"同义标签爆炸"污染薄弱点统计;
- **微调语料囤积**(人工校对前快照 AI 原始识别到 ai_snapshot_json),为将来自有模型留了数据;
- 视觉拆题按 bbox 生成**单题裁切图**,has_figure 全链路贯穿,解决了含图题这一硬骨头;
- prompt 注入有意识防护(BEGIN/END 隔离块 + 输入截断),并诚实注明局限。

### 4. 安全与成本底线意识好
httpOnly cookie、19 项越权回归测试、AI 配额双护栏(单用户 30/天 + 全站 200/天)、注册邀请码、JWT_SECRET 未设置拒绝启动、每日 VACUUM 备份、CORS 白名单。个人项目里这个安全自觉是罕见的。

### 5. 工程哲学务实,文档纪律强
技术选型全部成熟保守(Express/SQLite/Vite),没有过度设计;README-driven,PROGRESS 作为唯一真相源持续维护;多轮自我审计文档,连"疏忽清单"都写得诚实。这套纪律是项目能持续演进的底座。

## 二、缺点与不足

### 产品层

| # | 问题 | 说明 |
|---|------|------|
| P1 | **愿景与实现断层大** | README 定义的七大能力中,Memory/Cognition/Knowledge/Decision/Review 目前只是概念。代码里对应的仅是 mistake_records 的标签计数 + learning_path_items 的规则推荐。"认知模型""知识图谱""三层知识树"没有任何数据结构支撑。README 里承诺的 Cache 和 RAG 也未实现。 |
| P2 | **命脉功能缺位** | 自家战略文档认定"PDF 导入质量 + 重做机制是命脉,其它都是配菜",但重做机制至今未实现(《重做机制开发计划.md》待评审)。差异化闭环缺最后一环:学生错了之后,没有"遮蔽重做→亲眼看到分数变化"的动作。 |
| P3 | **零真实用户、零埋点** | 到 v2.2 仍没有一个真实学生的完整使用记录,也没有任何事件埋点——不知道用户在导入、做题、评阅哪一步流失。这是当前最大的风险,大于任何代码问题。 |
| P4 | **AI 体验有硬伤:无流式输出** | 评阅/解析全部等完整 JSON 返回,学生提交作答后要面对 20-60 秒的转圈(aiTimeoutMs=60s)。移动端弱网下体感更差。这是"学生会不会用第二次"的直接杀手,但现有审计文档均未提及。 |
| P5 | **配额模型与核心场景冲突** | 按"次"计配额(30 次/天),而导入一份 20 页 PDF 就要 20+ 次视觉调用,做一套 19 题卷再花 19 次评阅——一个认真用的学生一天就顶格。按次而非按 token 成本计,粒度错了。 |
| P6 | **账号体系不适配目标人群** | 昵称+密码、无自助找回(仅管理员脚本)。高中生必然忘密码;国内正式运营终究要手机号/微信。托管在 Vercel/Railway 对大陆学生的可达性是硬伤(已在战略审计中识别,未解决)。 |

### 工程层

| # | 问题 | 说明 |
|---|------|------|
| E1 | **巨石路由,无分层** | routes.js 1403 行 + routes-import.js 1402 行,HTTP 解析、业务逻辑、SQL、AI 调用全部揉在路由处理器里。加任何功能都在往两个大文件里堆,回归风险持续上升。 |
| E2 | **AI 输出无 schema 校验** | ai.js 的 extractJson 只做 JSON.parse,不校验字段形状。模型少给一个字段、score 给成字符串、step_breakdown 给成对象,都会直接入库并返给前端。数据质量无闸门,而这些数据正是将来认知模型的原料。 |
| E3 | **长任务跑在 HTTP 请求里** | `/import/pipeline/tasks/:taskId/process-all` 在一个请求内逐页调视觉模型,20 页 PDF = 一个数分钟的同步请求。无任务队列、无进度推送(前端只能轮询)、平台网关超时或重新部署即中途失败,且无断点标记。 |
| E4 | **同步 SQLite + 内存态限流** | node:sqlite 的 DatabaseSync 每次查询都阻塞事件循环(数据量增长后写放大);rate-limiter 是进程内存版,重启清零、多实例无效。单进程单点。 |
| E5 | **上传走 base64 JSON** | PDF/图片以 base64 塞进 30MB JSON body,内存放大约 33%,大文件全量驻留内存。应改 multipart 流式。 |
| E6 | **无类型、无 lint、测试无框架、无 CI** | 全 JS 无类型标注;无 ESLint/Prettier;测试是手写 .mjs 脚本(质量其实不错但无断言框架、无覆盖率),没有 CI——守护栏全靠本地手动跑。前端零测试。 |
| E7 | **数据模型的查询短板** | knowledge_tags_json 等 JSON-in-TEXT 列无法索引,薄弱点统计靠全表扫 + 应用层解析;主表(questions/attempts/mistakes)缺复合索引(v2.1 审计已列为 P1,仍未落)。 |
| E8 | **前端样式与状态管理粗放** | 1345 行单文件 styles.css,已出现"移动端被主题重构回归挤压,靠尾部重申单列修复"这类级联事故;design tokens 只有 35 行雏形。user 状态在 App.jsx 手工管理,与 React Query 双轨。 |
| E9 | **可观测性为零** | 无结构化日志(仅 console.error)、无错误监控、无接口耗时指标。线上出错只能等用户截图。 |
| E10 | **仓库卫生** | 根目录残留历史分裂的 studyoo.db + uploads(2026-07-12 已知,仍"待决定");.obsidian、未命名.canvas 混入版本库;十余份 md 文档散在根目录无归档结构。 |

### 运维层

- **O1** 备份仍只落在同一 /data 卷内,卷级故障 = 全部用户数据消失(已识别,未解决)。
- **O2** 密码无强度要求;CORS/环境变量靠手工配置,换域名会静默炸。

## 三、优化重构升级方案

**总原则:用户验证 > 命脉功能 > 体验 > 工程重构 > 规模化。**
零用户阶段最大的风险是方向错,不是代码丑。因此重构安排在拿到真实使用数据、补完差异化闭环之后;所有规模化投入(换数据库、国内迁移)以真实留存数据为触发条件,不提前做。

### 阶段 0:守住底线 + 拿到真实数据(本周,约 3 天)

| 任务 | 做法 | 验收 | 状态 |
|------|------|------|------|
| 异地备份 | 复用 backup.js,备份完成后推送 Cloudflare R2(S3 兼容,免费额度足够),保留 30 份 | 删除本地备份后能从 R2 恢复 | ✅ 2026-07-15(手写 SigV4,配置 BACKUP_S3_* 即启用) |
| 错误监控 | 前后端接 Sentry(各约 10 行);后端换 pino 结构化日志 | 手动抛错能在 Sentry 看到 | ⏳ 需 Sentry 账号 DSN,待 GinEXX 提供后接入 |
| 最小埋点 | 自建 events 表 + `POST /events`,只埋 7 个事件:注册、导入开始、导入成功、开始做题、提交作答、完成订正、次日回访 | 能导出一张漏斗表 | ✅ 2026-07-15(`npm run report:funnel`) |
| 密码找回兜底 | 注册时可选填联系方式;登录页放"忘记密码→联系邀请人"入口 | 流程可走通 | ✅ 2026-07-15 |
| **真实学生内测** | 发 3-5 个邀请码,让学生导入自己最近一张月考卷,完整走一遍链路 | 每人一份漏斗记录 + 访谈笔记 | ⏳ 待 GinEXX 发码(工程侧已就绪) |

阶段 0 结束时应能回答:导入卡在哪、评阅信不信、会不会第二天回来。这些答案决定阶段 1 的排序微调。

### 阶段 1:补完差异化闭环(2-3 周)

1. **重做机制**(按《重做机制开发计划.md》M1→M4,约 2 天起步)——命脉功能,最高优先。✅ 2026-07-15 完成(26 项 e2e 测试 + 浏览器实测)
2. **AI 评阅流式输出(SSE)**:callChat 增加 stream 模式,评阅的 feedback_text 边生成边渲染;结构化字段(score/step_breakdown)在流末尾聚合。体感等待从 40s 降到首 token 3s 内。这是移动端留存的关键一役。⏳ 待做(与 JSON 输出格式有耦合,需单独设计流式协议)
3. **AI 输出 schema 校验**:为 evaluatePracticeAttempt 等每个 AI 函数定义 schema,parse 失败自动带错误信息重试一次,再失败才报 AI_SERVICE_ERROR。✅ 2026-07-15 完成(ai-schemas.js 手写校验器,零新依赖;视觉识别按设计走人工校对兜底不强校验)
4. **配额改按成本计**:ai_usage 记录 token 数,限额改为每日 token 预算。✅ 2026-07-15 完成(token 计量 + 双层预算护栏;评阅缓存 practice_evaluation_cache 项目原已实现)
5. **能力地图 v1**:知识点 × 掌握度一张可视化,订正/重做成功后掌握度肉眼可见地变化。✅ 2026-07-15 完成(个人页三级掌握度配色 + 订正率/重做通过率)
6. **PWA**:manifest + service worker,可添加到主屏幕,静态资源离线缓存。✅ 2026-07-15 完成

### 阶段 2:工程重构(3-4 周,可与阶段 1 尾部并行)

后端(目标:单文件 <300 行,新人能在一天内看懂结构):

```
backend/src/
  routes/        # 只做 HTTP:参数校验、调 service、返回
  services/      # 业务逻辑:practice、import、review、recommend、quota
  repos/         # 全部 SQL 收口,JSON 列的序列化/反序列化只在这层发生
  ai/            # 调用 + zod schema + prompt 模板分文件
  jobs/          # 任务队列 runner
```

- **拆分顺序**:先 repos(纯搬 SQL,行为不变,风险最低)→ 再 services → 最后瘦身 routes。每步跑全量守护栏测试。
- **长任务改队列**:建 jobs 表(id/type/payload/status/progress/error),process-all 改为"创建任务立即返回",进程内 job runner 逐页处理、逐页更新 progress(天然获得断点续传:失败从未完成页继续);前端 SSE 或轮询进度。不引入 Redis——SQLite 表驱动足够支撑到千级用户。
- **上传改 multipart**(multer),PDF 流式落盘,撤掉 30MB JSON 特例。
- **测试框架化**:vitest + supertest 承接现有手写脚本的用例;GitHub Actions 每次 push 跑测试。✅ CI 已就绪 2026-07-15(.github/workflows/ci.yml 跑 5 套测试 + 前端构建;vitest 迁移待做,现有脚本已能在 CI 中守护)
- **补复合索引**(审计 P1):attempts(user_id, question_id)、mistake_records(user_id, mastery_status)、questions(user_id, status, created_at) 等。✅ 2026-07-15 完成(6 条)
- **类型渐进**:不做全量 TS 重写;新文件用 TS,存量文件先靠 zod(边界校验)+ JSDoc(编辑器提示)守住,repos/services 拆出时顺手迁移。

前端:

- styles.css 按页面/组件拆分,完善 tokens(colors.css 扩展为 spacing/typography/radius),CSS 层面立规矩防再次发生"主题重构挤垮移动端"式回归;
- api.js 扩展为统一 client(错误分类、401 全局跳登录、toast 收口),user 状态并入 React Query;
- 关键流组件测试(vitest + testing-library):做题提交、导入校对、重做。

仓库卫生(半天):文档归入 docs/(按 audits/plans/specs 分类),删除根目录历史 studyoo.db 与 uploads(先确认无独有数据),.obsidian 与 tmp/ 入 .gitignore。

### 阶段 3:规模化升级(触发条件:内测留存达标、决定认真运营;1-2 个月)

1. **数据层**:SQLite → Postgres(自托管或 Neon/Supabase),标签 JSON 列拆关系表;图片/PDF 迁对象存储。repos 层已收口 SQL,迁移成本可控——这正是阶段 2 分层的回报。
2. **国内落地**:自定义域名实测大陆可达性 → 国内云 + ICP 备案 → 手机号/微信登录替换昵称密码。合规同步处理(未成年人隐私政策、家长同意)。
3. **认知模型 v1 落地**:从"标签计数"升级为真正的掌握度模型——每个(学生, 知识点)维护记忆状态,用 FSRS 类间隔重复算法驱动 Review 排程;每次评阅/重做结果作为观测更新状态。README 的 Memory/Cognition/Review 三个能力由此从概念变成数据结构。
4. **RAG**:课标、教材目录、考点结构建知识库,解析与评阅前检索注入——兑现 README 承诺,同时降低模型幻觉。
5. **多学科扩展**:数学链路验证成立后,物理/化学复用同一套导入-练习-重做管线。

### 明确不做/缓做

- 不现在换数据库、不全量重写 TS、不上微服务/Redis/K8s——当前规模全是负资产;
- 不做开放注册与增长投放,直到评阅可信度与次日回访在内测中得到验证;
- 原生 App 不做,PWA → (验证后)微信 H5/小程序。

### 里程碑衡量

| 阶段 | 核心指标 |
|------|----------|
| 0 | ≥3 名真实学生完整走完"导入→做题→评阅"漏斗,拿到流失点数据 |
| 1 | 评阅首 token <3s;重做闭环上线;内测学生 7 日留存 ≥40% |
| 2 | CI 绿灯常态化;最大源文件 <300 行;导入 20 页 PDF 全程有进度、可断点续传 |
| 3 | 国内网络 P95 打开 <2s;认知模型驱动的 Review 排程上线 |
