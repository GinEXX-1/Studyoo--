## 1. 第一阶段：地基（对应整体计划第1-2周）

### 1.1 数据库设计与建表

按 `API契约.md` 第1节的数据模型，设计对应的数据库表结构（User / Question / Answer / MistakeRecord / LearningPathItem）。

- 输出一份 `数据库设计.md`，列出每张表的字段、类型、外键关系，供其他人对照
- 建表脚本要能一键跑起来（迁移脚本或建表SQL），不要手动在数据库里点出来的表结构，否则没法复现

### 1.2 账号模块（契约第2节）

实现：

- `POST /auth/register`
- `POST /auth/login`
- `GET /users/me`
- `PATCH /users/me`

验收标准：用Postman或类似工具，真实注册一个账号、登录拿到token、用token访问 `/users/me` 能拿到刚注册的信息。这一步必须自己手动验证一遍，不能只看代码逻辑"应该没问题"。

### 1.3 待拍板事项确认

契约第6节的4个问题，在开始写答疑模块之前必须有明确答案，写回契约文档：

- AI服务商选型、超时时间、重试次数
- 图片存储方案
- 限流规则（单用户每日提问上限）
- token过期时间与处理方式

## 2. 第二阶段：智能答疑模块（对应整体计划第3-5周，核心优先级最高）

这是整个产品最核心的差异化功能，重点做透，不要赶进度牺牲质量。

### 2.1 两种提问模式

按契约第3节，务必区分 `mode` 字段的两种行为：

- `solve_from_scratch`：从零讲解，先给 `hint_text` 引导思路，用户主动请求才给完整解答
- `deepen_official_answer`：用户已有官方答案，直接返回 `step_breakdown`（分步骤解释官方答案每一步在做什么、为什么），不藏答案

这两种模式在你写AI调用的系统提示词（prompt）时，应该是两套不同的prompt模板，不要用一套prompt靠参数硬凑出两种效果，容易两头都做不好。

### 2.2 数学公式LaTeX约定（重要，容易被忽略）

调用AI模型的系统提示词里，必须显式要求：涉及数学表达式一律用标准LaTeX语法，行内用 `$...$`，独立公式用 `$$...$$`。

验收标准：真实提交一道包含分式、根号、求和符号的数学题，检查AI返回的文本里公式是否被正确包裹在定界符里，而不是纯文本表达式（比如 `x^2+y^2` 这种没有 `$` 包裹的写法是不合格的）。

### 2.3 图片输入

按契约3.2，接入OCR/多模态模型直接处理图片转文字+解答，不要额外自建图片存储服务（除非第1.3节的待拍板事项里已经决定要做）。

### 2.4 追问接口

契约3.5，注意追问需要带上下文（原题目+已给出的讲解），不能让AI在追问时"失忆"，重新从零回答。

## 3. 第三阶段：错题本模块（对应第5-7周，可与答疑模块收尾并行）

按契约第4节：

- 错题判定规则第一版按契约里的简化规则来——用户点击过"直接看答案"即计入错题本，不要自己发明更复杂的判定逻辑
- `GET /mistakes/stats` 的错误率统计，要能真实反映一段时间内（比如近7天/近30天，自己定一个默认值并写清楚）的知识点错误情况

---

## 4. 第四阶段：个性化学习路径（对应第7-9周，依赖前面模块数据）

按契约第5节：

- 第一版就是规则引擎：知识点错误率超过阈值（默认50%，可配置）→ 生成一条推荐
- 用定时任务（比如每天跑一次）计算，不需要做成实时触发，不要过度设计

## 5. 每完成一个接口，必须做的事（贯穿始终）

1. 对照 `API契约.md` 的联调进度表，把对应行的"后端完成"打勾
2. commit信息用中文写清楚：改了什么、为什么改、影响范围（例如："实现 deepen_official_answer 模式的答疑接口，涉及 questions 表新增 mode 字段，不影响已有的 solve_from_scratch 逻辑"）
3. 如果这次改动涉及到契约文档里字段的变化，先改文档再改代码，并在commit信息里注明"同步了API契约的XX变更"
4. 告诉GinEXX一句"现在可以怎么验证"，比如"现在可以在网页里提交一道题，选深化答案模式，应该能看到分步骤讲解"——不要只说"做完了"

## 6. 与Trae的接口边界

- 你只负责返回契约里定义好的JSON结构，不负责任何展示逻辑
- 如果Trae反馈某个字段拿不到、格式不对，先检查是不是双方对契约的理解有分歧，而不是各自私下改一套
