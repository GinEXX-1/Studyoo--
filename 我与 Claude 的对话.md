## 问题一：前后端连不上，前端全是假数据

**根本原因**：先做前端UI、用假数据填充，打算"以后"接后端——这个"以后"经常永远不会来，因为一旦UI看起来能跑，改起来的意愿就低了，而真正对接API往往涉及大量边界情况修正。

**解法：改成"垂直切片"开发，而不是"水平分层"开发**

不要这样做：

```
第1-2周：搭完整前端（所有页面，假数据）
第3-4周：搭完整后端
第5周：对接（灾难现场）
```

而要这样做：

```
第1周：选一个最小功能（比如"提交一道题→AI返回讲解"）
       前端一个输入框+一个结果展示区，后端一个真实接口，从第一天就是真连接
第2周：这个功能能跑通、你能亲自在浏览器里用真实数据点一遍，才算完成
第3周：加下一个功能（错题保存），同样是端到端打通
```

关键原则：**任何时候都不允许"前端先用假数据，以后再接"这种承诺**。哪怕最简陋，第一版就必须是真数据在真的跑。这样"连不上"这件事会在第1周就暴露，而不是第5周。

**给agent的具体要求（这句话可以直接说给Codex/Trae）**：

> "不要用mock data，直接调用真实的后端接口。如果后端还没有这个接口，先把后端接口写出来并跑通，再写前端。"

## 问题二：不知道agent改了哪里

这是更根本的问题——你现在其实是在"盲开"，agent的输出你无法验证。解法不是让你去读代码（你不需要变成程序员），而是**建立一个每次都能让你看懂"发生了什么"的机制**。

**具体做法：**

1. **强制用Git，每次改动都是一个可读的commit** 每次让agent做完一件事后，要求它：
    
    > "把这次改动commit，并且用中文写清楚：改了哪些文件、为什么改、有没有影响到其他功能。"
    
    你不需要看代码本身，只需要看这段中文描述是否合理。如果agent说不清楚自己改了什么，这本身就是危险信号。
    
2. **每次改动后，要求agent给你一个"如何验证"的步骤** 不是"我改好了"，而是："我改好了，你现在打开网页，点X按钮，应该看到Y结果"。你亲自动手验证一遍，而不是听agent的一面之词说"已修复"。
    
3. **保持一份"当前状态文档"，agent必须同步更新** 建一个简单的 `PROGRESS.md`，里面写清楚：
    
    - 现在哪些功能是真实可用的
    - 哪些还是占位/未完成
    - 后端有哪些接口、分别做什么
    
    每次agent做完事，要求它更新这份文档。这份文档比代码本身对你更重要，因为它是你唯一能读懂的"真相来源"。
    
4. **小步提交，别让一次任务改动太多文件** 一次只让agent做一件具体的事（"加一个提交题目的接口"），而不是"把答疑模块做完"。改动范围越大，你越没法追踪，出问题也越难回滚。出问题时可以直接回退到上一个commit，这是你唯一的"后悔药"，前提是你有commit习惯。
