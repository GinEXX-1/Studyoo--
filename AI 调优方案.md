# AI 调优方案（替代"训练模型"的务实路线）

制定日期：2026-07-12

---

## 先说结论：现阶段不训练模型

"训练 AI 模型使其贴合项目"有三条路，成本和收益差异巨大：

| 路线 | 成本 | 现阶段收益 |
|---|---|---|
| 微调（fine-tune）BigModel 模型 | 需数千条高质量标注样本 + 训练/托管费用 | 低——当前质量问题不在模型权重 |
| 提示词工程 + 评测集 | 几乎为零 | **高——当前 90% 的质量问题源自 prompt 与上下文** |
| RAG（把课标/知识点目录喂给模型） | 低 | 中——标签规范化、评阅口径统一 |

证据：2026-07-12 仅通过重写拆题 prompt（不改模型），PDF 整页"一锅炖"问题就从"整页并为一题"变成"一页正确拆出 8 题、7 题置信度 0.95"。**先把免费的杠杆用完，再谈花钱的。**

## 三层调优路线

### 第一层：提示词（已开始，持续迭代）

已完成（v2.1）：
- 拆题 prompt 重写：题号边界规则、禁止合并、选项归属、截断处理、答案页判定（backend/src/ai.js `recognizePageQuestions`）。
- 新增单题精细识别 prompt（`recognizeSingleQuestion`），配合裁切图重识别。

待做：
- 评阅 prompt 增加评分锚点（什么样的作答是 60 分/80 分/100 分，各给一个具体例子），减少打分抖动。
- 重做对比评阅 prompt（见《重做机制开发计划.md》M2）。
- 所有 prompt 集中到独立文件（如 `backend/src/prompts.js`），每个 prompt 带版本号，改动可追溯。

### 第二层：评测集（本方案的核心交付）

现有 `验收题集.md` 只有 4 题且靠人肉验证。升级为**可重复运行的评测脚本**：

`backend/scripts/eval-ai.mjs`（建议新增）设计：

1. **拆题评测**（5 份标注页面）：输入页面图，断言识别题数 == 金标准题数、题号集合一致。输出准确率。
2. **评阅一致性评测**（10 条金标准作答）：每条含（题目、学生作答、人工评定分数区间、必须指出的缺口关键词）。断言 AI 分数落在区间内、feedback 命中关键词。同一条跑 3 次看抖动。
3. **公式转写评测**：含分式/根号/求和的题目，断言输出含合法 LaTeX（`$...$` 配对、无裸 Unicode 数学符号）。
4. 每次改 prompt 后跑一遍，结果追加到 `eval-history.md`，**prompt 改动没有评测对比就不合入**。

成本控制：全量评测约 25-30 次 AI 调用，建议用独立 API key 或在 AI_DAILY_LIMIT 外单独记账。

### 第三层：RAG / 上下文注入（部分已有，扩展）

已有：canonical 知识点目录随每次调用注入（`canonicalTagHint`）。

扩展方向（按优先级）：
1. **评阅注入题目 AI 档案**：评阅时把该题的 `ai_question_profiles`（核心思想、常见错误）拼进 prompt——AI 评阅从"裸看题"变成"带着命题意图看作答"。改动极小，收益直接。
2. 课标知识树（README 规划的 Knowledge 能力）建成后，按学科注入对应章节要求。
3. 学生认知画像（Cognition）注入：把该生该知识点的历史错误模式给 AI，实现"因人评阅"。

## 为未来的微调囤数据（现在就做，零成本）

真到需要微调那天，最贵的是数据。而 Studyoo 的产品流程天然在生产标注数据：

- **PDF 校对 diff**：`question_candidates` 里 AI 识别原文 vs 人工校对后的终稿，就是现成的"识别纠错对"。建议在 PATCH 校对接口里把修改前的字段快照存一列 `pre_review_snapshot_json`（一行代码），日积月累就是拆题/转写微调集。
- **评阅申诉**：将来若加"我觉得评分不对"按钮，收集的就是评阅微调集。

数据攒到 1000+ 条、且评测集证明 prompt 已到天花板时，再评估 BigModel 微调或换模型。

## 模型选型现状与建议

| 用途 | 当前配置 | 建议 |
|---|---|---|
| 文本（评阅/解析/组卷） | glm-5.2（.env AI_MODEL） | 保持；评测集跑分后可对比 flash 版省成本 |
| 视觉（拆题/识图） | AI_VISION_MODEL | 拆题质量优先，不建议降级 |
| 超时 | 60s | 视觉单页实测 ~44s，接近上限；建议视觉调用单独放宽到 90s |
