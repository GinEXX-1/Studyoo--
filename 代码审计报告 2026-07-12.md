# Studyoo 代码审计报告

审计日期：2026-07-12
审计范围：backend/src 全部源码、frontend/src 关键路径、配置与数据文件
说明：本报告针对**代码实现**，与 2026-07-05 的《项目审计报告.md》（针对文档与契约）互补，不替代该文件。

---

## 总体结论

代码基础素质不错：全线参数化 SQL、bcrypt 密码哈希、httpOnly cookie、JWT_SECRET 启动强校验、统一错误码、关键写入有事务、AI prompt 有注入缓解措施。**单人本地使用是安全的。**

但当前实现隐含一个假设——"所有用户互相信任"。一旦部署到多人环境，存在 3 个高危问题：公共题库可被任意登录用户改写（投毒）、用户间数据无隔离（可看/可下载他人上传的试卷 PDF）、PDF 导入流水线完全绕过 AI 配额。另有一个数据架构问题正在实际造成浪费：PDF 原文以 base64 双份存进 SQLite。

---

## 高危（部署给第二个用户之前必须修）

### 1. 任何登录用户可改写公共题库（数据投毒）

位置：[routes.js:414](backend/src/routes.js#L414) `POST /exam/ingest/manual`

- `paper.id` 和每道题的 `id` 允许调用方任意指定，且 `ON CONFLICT(id) DO UPDATE` 无所有权检查。
- 后果：任何注册用户可以指定 `id: "exam-2026-national-1-01"` 覆盖内置 2026 年全国 1 卷的题干和参考答案，同时 `DELETE FROM ai_question_profiles` 会清掉该题的 AI 档案。所有用户的练习、评阅都会被污染（评阅以 official_answer_text 为准）。
- 建议：导入的 paper/question 强制归属 `owner_user_id`；更新时校验 `owner_user_id = req.user.id`；内置种子数据（owner 为 NULL）禁止通过 API 更新。

### 2. 用户间数据无隔离，可下载他人的原卷 PDF

位置：[routes.js:329-357](backend/src/routes.js#L329)、[server.js:41](backend/src/server.js#L41)

- `GET /exam/papers` 返回**所有**试卷（含其他用户 PDF 导入的个人试卷及其 `pdf_url`），`owner_user_id` 字段存了但从未参与过滤；`GET /exam/papers/:id/questions`、`GET /exam/questions/:id` 同样无所有权检查。
- `/uploads` 静态目录完全无鉴权：拿到 URL（上一条正好会泄露）即可直接下载任何用户上传的 PDF 和页面图，甚至无需登录。
- `practice_questions` 也是全局共享：A 导入的题 B 能刷到（`/practice/questions/current` 按 id 顺序遍历全表）。
- 对比：`questions`/`mistakes`/`learning-path`/`import/pipeline` 都正确做了 `user_id` 过滤——契约中"所有权鉴权"规则只落实了一半。
- 建议：exam/practice/collections 查询统一加 `(owner_user_id = ? OR owner_user_id IS NULL)`；`/uploads` 加 `requireAuth` 并校验文件归属（或至少加鉴权）。

### 3. PDF 导入流水线完全绕过 AI 每日配额

位置：[routes-import.js:270-464](backend/src/routes-import.js#L270)（process、process-all、re-recognize）

- 这三个接口直接调用视觉模型 `recognizePageQuestions`，不经过 `withAiQuota`。一个 15 页 PDF 的 process-all 就是 15 次视觉模型调用，可无限重复触发。恰恰是单次成本最高的接口没有配额保护。
- PROGRESS.md 中"全路由 save-first 配额保护"的描述与事实不符。
- 建议：每页识别计 1 次配额（或给导入流水线单独的每日页数限额），并修正 PROGRESS.md。

## 中危（建议近期处理）

### 4. PDF 原文 base64 存进 SQLite，双份存储且从不读取

位置：[routes-import.js:210-215](backend/src/routes-import.js#L210)、[migrate-v2.js:29](backend/src/migrate-v2.js#L29)

- `import_tasks.pdf_data_base64` 存了整个 PDF（当前 backend 库 11 个任务约 19MB、根目录库 4 个任务约 13MB），而同一个 PDF 已经写入 uploads 目录——纯双份。全代码库没有任何地方读取这一列。
- 更糟的是任务列表接口用 `SELECT * FROM import_tasks ... LIMIT 30`，每次列表请求会把最多 30 × 27MB 的 base64 拉进内存再丢弃。
- 建议：删除该列（迁移置 NULL 即可立刻回收），列表查询改为显式列名。

### 5. 同一项目两套数据库、两套 uploads 目录

- `DATABASE_PATH=./studyoo.db` 和 `UPLOAD_DIR=./uploads` 都是相对启动目录的路径。现状：根目录 studyoo.db（17MB）+ uploads（40MB）与 backend/studyoo.db（24MB）+ uploads（84MB）并存，取决于从哪里启动进程，数据已经实际分裂。
- 建议：在 config.js 中像处理 .env 那样用 `__dirname` 把两者 resolve 到 backend 目录；确认哪份是真数据后清理另一份。

### 6. KaTeX 渲染失败时把原文当 HTML 注入（潜在 XSS）

位置：[MathText.jsx:14](frontend/src/components/MathText.jsx#L14)

- `renderFormula` 的 catch 分支返回 `{ __html: source }`，源文本未转义直接进 `dangerouslySetInnerHTML`。公式内容来自 AI 输出和（可被他人改写的，见问题 1）题库内容。`throwOnError: false` 下触发概率低，但修复只需一行：fallback 改为渲染纯文本节点。

### 7. PATCH 更新不校验影响行数

位置：[routes.js:1108](backend/src/routes.js#L1108)、[routes.js:1214](backend/src/routes.js#L1214)

- `PATCH /mistakes/:id`、`PATCH /learning-path/:id` 对不存在（或不属于自己）的 id 也返回 200 成功。越权是防住了，但调用方无法发现 id 传错。建议检查 `.run()` 返回的 changes，为 0 时返回 404。

### 8. 所有"今天"都是 UTC 日期

位置：[routes.js:42](backend/src/routes.js#L42)、[routes-review.js:81](backend/src/routes-review.js#L81) 等

- AI 配额、间隔复习的 scheduled_date、"今日待复习"判定都用 `toISOString().slice(0,10)`，即 UTC 日。对中国用户：配额每天早上 8 点才重置；晚上 8 点后做错的题，"当天复习任务"会被记到"明天"。建议统一用本地时区（或明确固定 Asia/Shanghai）生成日期字符串。

### 9. 开发环境暴露面

- `vite --host 0.0.0.0`（frontend/package.json）让前端 dev server 对整个局域网开放。
- `PDF_RENDER_ERROR` 把 `err.message`（含服务器绝对路径）原样返回给客户端（[routes-import.js:199](backend/src/routes-import.js#L199)）。
- 认证限速是纯内存 Map，重启清零，register/login 共享同一计数键。本地开发都可接受，部署前需要处理。

## 做得好的地方

- SQL 全部参数化，未发现注入点（动态表名仅用于硬编码迁移，已有注释说明）。
- JWT_SECRET 未配置直接拒绝启动；密码 bcrypt；cookie httpOnly + sameSite strict。
- AI prompt 对用户输入做了定界（BEGIN/END 块）与截断，并在注释中如实说明该防护的局限。
- AI 配额"先扣后调、失败回滚"的设计（在覆盖到的路由上）是对的。
- 多步写入普遍用了 BEGIN/COMMIT/ROLLBACK；process-all 中还正确处理了"事务是否已开启"的状态。
- 错误响应结构统一（success/data/message/error_code），前端契约清晰。
