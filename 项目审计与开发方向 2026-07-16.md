## 二、技术层审计

### 做得好的(保持)

1. **数据护栏体系是同规模项目里的第一档**:越权测试、AI 输出 schema 闸门、token 双层预算、评分锚点、本地+异地双备份、httpOnly cookie。这套底线让"敢让真学生用"在工程上完全成立。
2. **规则驱动推荐的克制是对的**:routes-recommend.js 明确注释"AI 仅生成解释文案,不决定调度优先级"——可解释、可调试、零幻觉,这正确贯彻了"AI 在底层"的产品原则。
3. **微调语料囤积(ai_snapshot_json)** 继续在积累,这是将来最值钱的数据资产之一。

### 技术债排序(按"拖着不修的代价增速"排,非严重度)

| # | 问题 | 为什么现在要动 |
|---|------|----------------|
| T1 | **评阅无流式输出**。学生提交作答后面对最长 60 秒转圈(aiTimeoutMs=60s),移动端弱网更差 | 这是"学生会不会用第二次"的直接杀手,连续两轮审计标为留存关键。且 `/admin/stream` 已验证 SSE 在本栈可行,边际成本已降到最低 |
| T2 | **巨石路由在膨胀**。routes.js 1689 行 + routes-import.js 1411 行 + db.js 1027 行;PR #3 又新增 4 个平铺 routes-* 文件,分层缺失下每个新功能都在加宽回归面 | 重构成本随行数超线性增长,再拖一个版本,repos 抽取的"纯搬 SQL"窗口会关闭 |
| T3 | **长任务仍在 HTTP 请求里**。20 页 PDF = 一个数分钟同步请求,平台网关超时/重新部署即中途失败,无断点 | 这是导入(命脉功能之一)的可靠性天花板,真实内测一旦开始,这里最容易出"学生第一次用就失败"的事故 |
| T4 | **可观测性为零**。无结构化日志、无 Sentry;内测期间线上出错只能等学生截图 | 内测前必须就位,否则漏斗数据只能告诉你"流失在哪一步",不能告诉你"为什么" |
| T5 | 上传走 base64 JSON(30MB 特例)、同步 SQLite、内存态限流 | 千级用户前不致命,维持原判:阶段 3 触发条件到了再动 |
| T6 | 前端零测试;styles.css 单文件 1522 行,已发生过一次"主题重构挤垮移动端"级联事故 | 与 T2 同批处理 |
| T7 | CI 分支过滤漏掉 `claude/**`;根目录 15+ 份 md 无归档,`.obsidian`/`未命名.canvas` 混入版本库 | 半天卫生活,顺手做 |

### 一个新增的技术层观察:硬编码的"知识图谱"

routes-recommend.js 里的 `knowledgeDependencies` 是每科 5-7 条边的**手写玩具级 DAG**(如 数学:基础运算→函数→导数→综合应用),`contextKeywordTags` 靠关键词包含来贴标签。作为 v0 的可解释起点这没问题,**但它现在被前端 LearningPathGraph 以"学习图谱"的名义可视化给了用户**。README 承诺的是"课程知识树/考试知识树/学生个人知识树"三层结构——玩具 DAG 撑不起这个叙事,而学生一眼能看出"数学只有 7 个节点"的图是假的。这不是代码问题,是**教研数据问题**,详见愿景层。

## 三、愿景层审计

### 愿景资产:这个项目最不可替代的东西

《我与 Claude 的对话.md》里沉淀的定位在反复审视后依然成立,且比大多数同类产品想得深:

- **对抗的是"理解权让渡"**,不是"不会做题"——模糊解析、充值网课、死刷海题,这些是真实敌人;
- **AI 像纸笔一样在底层**,追问做成有边界的分支而非无限聊天;
- **独有闭环**:导入自己的月考卷 → 拆题 → 做错 → 订正 → 遮蔽重做 → 亲眼看到 45→88。重做机制上线后,这个闭环**第一次在代码里完整存在了**。这是本项目迄今最重要的里程碑。

一句话定位维持不变且现在有了实现支撑:**"别的 App 帮你看懂题,Studyoo 逼你做对它。"**

### 愿景风险 V1:方向漂移的第一个信号——Discover「新发现」

PR #3 把「新发现」做进了主导航:全站共享题目的信息流 + 五星评分 + 收藏 + 管理员从网页抓题入库。需要直说:

1. **它是计划外功能**。《优化方案 07-15》的阶段 1/2 没有它;《下一阶段功能计划 07-14》明确写了共享池"藏着必须先澄清的决策点(审核/版权/质量/归属/AI 学习形态),不建议直接开工",而这五个决策点至今没有落成文档。
2. **它把产品往"内容社区"的方向拉**,而 README 的立身之本是"个人学习系统":学生和**自己的**试卷、**自己的**错题、**自己的**理解的关系。信息流+评分是供给侧逻辑,与"第一场景是做真题"存在张力——刷 feed 恰恰是"结束思考"式的交互。
3. 缓解因素:web 抓题限 admin、共享仍走 v2.2 的 owner 机制,风险有闸。但版权(admin 从网页抓的题再分发)与质量决策仍是空白。

**建议**:不必回滚,但要求它"降级为配菜"——内测期间从主导航移到题库二级入口(或 feature flag),把五个决策点补成一页决策文档后再决定是否升回一级。**内测学生的注意力应该 100% 落在"导入自己的卷子"这条主链路上,这也是漏斗数据不被污染的前提。**

### 愿景风险 V2:零用户第三次上榜

这是连续第三份战略文档把"找 3-5 个真实学生"列为第一优先级,而它仍未发生。必须指出一个模式:**每一轮"还没找学生"的时间,都被用来开发了更多功能(v2.2 拍照/共享、v2.3 十项、PR #3 Discover)。功能开发正在成为回避用户验证的舒适区。**

现在工程侧已经没有任何借口:邀请码✅、埋点✅、漏斗报表✅、admin 实时数据台✅、反馈通道✅、备份✅。产品也已越过"值得被用"的线(完整重做闭环 + 含图题 + 数学键盘 + PWA)。**当前每一行新代码的边际信息量,都低于一个真实学生的一次完整使用。**

### 愿景风险 V3:README 的"七能力"与实现的断层在固化

| README 能力 | 实现现状 |
|---|---|
| Practice | ✅ 完整(含重做闭环) |
| Parse | ✅ 可用(辅助入口) |
| Memory | 🟡 有学习历史表,无"理解如何形成"的记录 |
| Cognition | 🟡 标签计数 + 三级掌握度配色,不是认知模型 |
| Knowledge | 🔴 硬编码玩具 DAG 冒充三层知识树 |
| Decision | 🟡 规则推荐 v2,方向正确但输入贫瘠(依赖上面两项) |
| Review | 🟡 间隔复习链存在,"理解成长轨迹"没有 |

断层本身不可怕(v0.5/v1.0 本来就是远期),可怕的是**用可视化把概念"演"成已实现**(学习图谱页)。对内测学生,一个诚实的"能力地图 v1(仅覆盖已练知识点)"比一张华丽但虚假的全科图谱更符合"Understanding belongs to you"的品格。

## 四、下一步开发方向

### 总原则(较 07-15 方案仅一处修订)

07-15 的排序"用户验证 > 命脉功能 > 体验 > 工程重构 > 规模化"依然正确。修订是:**把"用户验证"从优先级第一升格为其它一切工作的前置闸门**——在拿到 ≥3 份真实漏斗数据之前,冻结一切新功能开发(bug 修复与下述 S1/S2 除外)。理由见 V2:排序没能阻止功能开发挤占验证,只有闸门能。

### 冲刺 S0(本周,非工程):内测启动

- 发 3-5 个邀请码,对象:自己 + 身边同学,任务:导入自己最近一张月考卷,完整走"导入→做题→评阅→订正→重做"。
- 每人收一份:漏斗记录(`npm run report:funnel`)+ 15 分钟访谈(三问:导入卡在哪?评阅信不信?明天还会打开吗?)。
- Discover 从主导航降级(1 小时改动),保证漏斗干净。
- 内测期间工程侧只做 S1/S2,随时待命修学生遇到的 bug——**修真实用户的 bug 优先于下面一切**。

### 冲刺 S1(与 S0 并行,约 3-4 天):评阅流式输出 + 内测可观测性

1. **SSE 评阅**(T1):复用 `/admin/stream` 已验证的模式。协议建议:评阅 prompt 拆两段——先流式输出 `feedback_text`(学生 3 秒内看到"你第 1 问的思路是对的…"开始逐字出现),流末尾再一次结构化调用(或同流的 JSON 尾块)出 score/step_breakdown,沿用 ai-schemas 校验。评阅缓存命中时直接整包返回。验收:首 token < 3s,弱网可降级回非流式。
2. **pino 结构化日志**(T4 前半,不等 Sentry DSN):请求日志 + AI 调用耗时/token + 错误堆栈,一天工作量。Sentry 拿到 DSN 后 10 行接入。
3. CI 分支补 `claude/**`(一行)。

### 冲刺 S2(内测进行中,约 1 周):导入可靠性

**长任务队列化**(T3,按 07-15 方案的 jobs 表设计):process-all 改"建任务即返回 + 进程内 runner 逐页处理 + 逐页落 progress",天然获得断点续传;前端进度条复用 SSE。这是内测中最可能炸的点,先于重构做。multipart 上传(T5 前半)若内测中出现大 PDF 失败再提级。

### 冲刺 S3(内测数据到手后,2-3 周):按数据修 + 启动分层重构

- 先按漏斗与访谈修真问题(此处无法预排,这正是内测的意义);
- 然后执行 07-15 阶段 2 的 repos 抽取(顺序不变:repos→services→瘦身 routes,每步全量跑守护栏)。**这次给它排上日期而不是"可与并行"——它已经让位两个版本了,T2 的膨胀速度说明不能再让。**

### 愿景层方向(与冲刺并行推进的三个决定)

**D1:给 Discover 一个明确的名分(一页决策文档)。** 回答 07-14 留下的五个决策点:审核机制、版权声明与下架、质量门槛、共享归属/撤回、"AI 学习"=RAG 而非训练。答案决定它是"共享题库的浏览入口"(收进题库页,安全)还是"社区"(需要完整的内容治理,建议至少推迟到 v1.0 后)。**在文档写出来之前,它保持二级入口。**

**D2:把"知识树"立项为教研工作,而不是继续当编程工作。** 路线:
1. 先只做数学一科:以课标 + 高考考纲为蓝本,人工整理一份权威知识点树数据文件(JSON,预计 100-200 节点,含父子关系与前置依赖)——这是《对话》里早已识别、被搁置至今的"地基";
2. 用它替换 `knowledgeDependencies` 硬编码 DAG,canonical 标签目录挂到树节点上;
3. 学生个人知识树 = 权威树 × 掌握度投影,能力地图与学习图谱都改为从真树渲染,只渲染有数据的子树(诚实原则);
4. 这份树同时是将来 RAG 的检索骨架与认知模型的状态空间——一份投入,三处回报。

**D3:认知模型 v1 的触发条件与形态提前定死,防止又被新功能插队。** 触发:内测 7 日留存 ≥40%(07-15 既定指标)。形态:(学生, 知识点) 二元组维护记忆状态,FSRS 类算法驱动 Review 排程,每次评阅/重做结果作为观测更新——README 的 Memory/Cognition/Review 由此从概念变成数据结构。在此之前,**不做**任何"看起来像认知模型"的展示层功能。

### 明确不做(维持 07-15 判断,新增两条)

- 不换数据库、不上 TS 全量重写、不上 Redis/微服务(维持);
- **新增:不再开发任何计划外的一级导航功能**——任何新入口先过"它帮助学生理解自己的卷子吗"这一问;
- **新增:不做全科知识树**——数学验证成立前,其它八科的硬编码 DAG 原样保留,不投入。

### 里程碑衡量(修订版)

| 冲刺 | 核心指标 |
|---|---|
| S0 | ≥3 名真实学生完整漏斗 + 访谈笔记(**唯一硬指标,其它全部为它让路**) |
| S1 | 评阅首 token < 3s;线上错误可在日志中定位 |
| S2 | 20 页 PDF 导入全程有进度、中断可续传 |
| S3 | 最大源文件 < 300 行;按内测数据修复的问题清单关闭 |
| D2 | 数学知识树数据文件合入,学习图谱渲染真树 |

## 五、一段话总结

**技术层**:v2.3 把安全、成本、数据护栏做到了个人项目的天花板,文档与代码一致性极高;下一步的技术主线只有三件事——评阅流式(留存)、导入队列化(可靠性)、后端分层(还债),其中 SSE 已在库内验证过,只差搬到主场景。**愿景层**:重做闭环的上线意味着"逼你做对它"第一次真实存在,产品已越过值得被用的线;当前最大的风险不是任何代码问题,而是"用功能开发回避用户验证"的模式已持续三个版本,以及 Discover 揭示的社区化漂移苗头。**下一步的本质是一次纪律行动:冻结新功能,发出邀请码,把三个学生的真实漏斗数据变成 S3 之后所有排期的输入;同时把知识树从编程问题重新定义为教研问题,立项数学一科的权威树——它是认知模型、RAG 和 README 三层知识树承诺共同的地基。**
